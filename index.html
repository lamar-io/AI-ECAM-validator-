Airbus Hackathon · Problem Statement 1

Intelligent Sensor Validation
for ECAM Systems

The Electronic Centralized Aircraft Monitor (ECAM) relies on raw sensor data to generate pilot alerts. When sensors produce inaccurate or inconsistent readings, the system generates false warnings — increasing pilot workload and reducing trust in real alerts. This project introduces an AI-based filtering layer that intercepts sensor data via the ARINC 429 bus, validates it in real time, and re-injects only trusted readings into the ECAM system.

Root Problem
⚠ Sensor Data Unreliability
ECAM sensors occasionally produce readings that are inaccurate or inconsistent with the aircraft's actual state. These faulty readings bypass any validation layer and directly trigger cockpit warnings — even when no real fault exists.

In high-stress flight scenarios, false alerts increase pilot cognitive load, create alarm fatigue, and risk diverting attention from actual emergencies. The Qantas Flight 32 incident generated over 80 ECAM alerts in one hour — a documented case of alert overload.
Proposed Solution
◈ AI-Based Filtering Pipeline
An intelligent preprocessing layer is inserted between the aircraft's physical sensors and the ECAM processing units. Operating on the ARINC 429 data bus — the standard avionics communication protocol in Airbus A320/A330/A340 — the pipeline applies a multi-stage AI stack to validate each sensor reading before it reaches the ECAM system.

Only readings that pass all validation stages are re-injected into the bus. Rejected readings are logged for maintenance review.
Sustainability & Safety Impact
✦ Flight Safety
Reduces false alerts reaching pilots, decreasing alarm fatigue during critical flight phases. Ensures pilots respond only to genuine system faults.
✦ Operational Efficiency
Fewer false maintenance groundings reduce unscheduled downtime and fuel waste from delayed or rerouted flights. Rejected sensor logs enable predictive maintenance.
✦ Environmental
More accurate engine parameter readings enable better fuel management decisions. Fewer unnecessary ground holds and diversions reduce per-flight emissions.

Digital Technologies

Technology Stack

Six verified technologies form the complete system. Click any card for technical depth.

01 / STAGE 1
Kalman Filter
Signal Smoothing · Preprocessing
A classical signal processing algorithm that estimates the true state of a sensor reading by combining the current noisy measurement with predictions from a physical motion model.
  • Operates on each incoming ARINC 429 data word before ML stages
  • Reduces instantaneous noise spikes without introducing lag
  • Particularly effective for pressure, temperature and RPM streams
  • Computationally lightweight — essential for onboard edge deployment
  • Provides a stable input signal that dramatically improves LSTM accuracy
Deterministic Lightweight Real-time
02 / STAGE 2
LSTM Neural Network
Temporal Anomaly Detection · ML
Long Short-Term Memory networks learn the expected time-series behavior of each sensor across different flight phases. Readings that deviate from learned temporal patterns are flagged as anomalies.
  • Trained on historical ECAM sensor sequences per flight phase
  • Captures long-range temporal dependencies (e.g., engine warm-up curves)
  • Outputs an anomaly probability score per time step
  • Handles multivariate sensor fusion — cross-validates related sensors
  • Model retrained periodically with new in-service data
Deep Learning Time-Series Supervised
03 / STAGE 3a
Isolation Forest
Statistical Outlier Detection · ML
An ensemble-based anomaly detection algorithm that isolates outlier sensor values by randomly partitioning the feature space. Anomalies are isolated faster because they differ from normal distribution.
  • Unsupervised — requires no labeled fault data to train
  • Assigns an anomaly score to each sensor reading
  • Highly efficient: O(n log n) training, O(log n) inference
  • Complements LSTM by catching sudden spike outliers
  • Works in parallel with Autoencoder for ensemble confidence
Unsupervised Ensemble No Labels Needed
03 / STAGE 3b
Autoencoder
Reconstruction-Based Detection · ML
A neural network trained to reconstruct normal sensor readings. Faulty readings produce high reconstruction error — indicating the input pattern was outside the normal distribution seen during training.
  • Trained exclusively on clean, normal sensor data
  • Reconstruction error threshold tuned per flight phase
  • Effective at detecting subtle, gradual sensor drift
  • Encoder output used as compressed feature vector for downstream stages
  • Combined with Isolation Forest for ensemble confidence scoring
Unsupervised Neural Network Drift Detection
04 / STAGE 4
Random Forest Classifier
Flight Phase Context · Decision
A multi-class classifier that identifies the current flight phase from combined sensor states. This allows the pipeline to apply correct phase-specific thresholds for each sensor, rather than global fixed limits.
  • Classifies: Taxi, Takeoff, Climb, Cruise, Descent, Landing
  • Each phase has distinct valid sensor ranges (e.g., EGT thresholds vary by phase)
  • High interpretability — decision trees can be audited for certification
  • Runs as final gate: reading must pass phase-specific threshold to proceed
  • Ensemble of 100+ decision trees for robust classification
Supervised Interpretable Multi-class
05 / BUS LAYER
ARINC 429
Avionics Data Bus · Integration
The verified standard avionics communication protocol used in Airbus A320, A330, and A340 aircraft. All sensor data from ECAM-monitored systems is transmitted via this unidirectional, point-to-point bus.
  • Unidirectional, point-to-point serial data bus (one transmitter, multiple receivers)
  • 32-bit word format with label, data, sign/status matrix, and parity fields
  • Operates at 12.5 or 100 kbit/s — appropriate for sensor data rates
  • Pipeline taps the bus passively, processes, then re-injects via a parallel bus
  • Standard for A310, A320, A330, A340 — directly applicable to this problem
Verified Standard Airbus Native Hardware Layer
